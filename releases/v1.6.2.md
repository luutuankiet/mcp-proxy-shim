# v1.6.2 — Restore retrieve_tools limit semantics after M1 dedup

**Released:** 2026-04-14
**Type:** Bug fix (backward-compatible, default-on with env knob)
**Audience:** Anyone running v1.6.1 against a fs-mcp fleet that sees fewer unique tools per `retrieve_tools(query, limit: N)` call than expected

---

## TL;DR

v1.6.1's M1 dedup folds byte-identical fs-mcp duplicates across hosts into a single canonical entry with `servers: [...]`. That's the intended behavior — but it quietly breaks the `limit` contract: when an agent calls `retrieve_tools("read_files", limit: 10)` against a 10-host fleet, upstream BM25 returns 10 byte-identical dupes, shim-side dedup folds them to 1, and the user gets **1 tool instead of the 10 unique-enough tools they asked for**. Host dupes consumed the upstream limit budget *before* dedup could fold them.

v1.6.2 fixes this by over-requesting the upstream call. The shim now passes `limit * multiplier` (default 3, cap 100) to mcpproxy-go, then slices shim-side to the original user limit after dedup. The user asks for 10 and always gets up to 10 unique tools.

- New pure-function helper `applyRetrieveOverRequest(body, multiplier)` in `src/middleware.ts` (stateless, testable)
- New env var `SHIM_RETRIEVE_OVERREQUEST_MULTIPLIER` (default `3`, floor `1`) — tune without redeploy if the fleet's duplication factor changes
- Wired at both entry points: `src/core.ts` (MCP `/mcp` path) and `src/daemon.ts` (daemon REST `/retrieve_tools`)
- Upstream hard cap of 100 honored per `mcpproxy-go internal/server/mcp.go:931-933`

---

## Why This Release Exists

v1.6.1 shipped dedup unconditionally (DECISION-012) so the tax-per-session went down dramatically. But the `limit` slice in `compactRetrieveTools` runs *after* dedup, which means:

1. Agent sends `limit: 10`
2. Shim forwards `limit: 10` unchanged to upstream mcpproxy-go
3. Upstream BM25 returns 10 results — all 10 happen to be byte-identical fs-mcp dupes across 10 hosts
4. Shim-side dedup folds all 10 into 1 canonical entry with `servers: [...]`
5. Shim-side slice caps at 10, but there's only 1 — the slice is a no-op
6. Agent sees **1 tool instead of 10**

The unique tools that would have differentiated the result set never made it into the upstream response because the duplicate-heavy entries consumed the limit budget first. Dedup should *surface* unique tools, not hide them behind a smaller unique count.

The fix: over-request upstream by a multiplier so there's enough raw headroom for dedup to fold duplicates AND leave `limit` unique entries behind. User-confirmed default multiplier: `3x`. At-cost: marginal upstream BM25 work (capped at 100 per mcpproxy-go's own validation).

## Before / After

**Before (v1.6.1) — `retrieve_tools("read files", limit: 10)` against 10-host fleet:**
```
upstream receives: limit=10
upstream returns: 10 byte-identical fs-mcp dupes
dedup folds:      10 → 1
shim slice to 10: no-op (only 1 left)
agent sees:       1 tool  ❌
```

**After (v1.6.2) — same call, default multiplier=3:**
```
upstream receives: limit=30  (10 * 3, capped at 100)
upstream returns: 30 results = 10 fs-mcp dupes + 20 other read_files-ish matches
dedup folds:      30 → ~21 (the 10 dupes collapse to 1, 20 others pass through)
shim slice to 10: returns 10 unique entries
agent sees:       10 tools  ✅
```

**Tuning:** fleets with heavier per-tool duplication (e.g. 20+ hosts with identical shims) can bump via `SHIM_RETRIEVE_OVERREQUEST_MULTIPLIER=5` without a redeploy. Fleets with lighter duplication can drop to `1` to disable the bump entirely.

## Verification

**Test suite:** 118 passed, 0 failed (`npm run test:daemon`). Three new E2E cases added:

| Test | Scenario | Assertion |
|---|---|---|
| retrieve_tools limit over-request | Default multiplier=3, `retrieve_tools(limit: 5)` | Upstream received `limit=15`; shim returned ≤ 5 entries |
| `SHIM_RETRIEVE_OVERREQUEST_MULTIPLIER` env override | Secondary daemon with `multiplier=5`, `retrieve_tools(limit: 10)` | Upstream received `limit=50` |
| 100-cap | Secondary daemon with `multiplier=10`, `retrieve_tools(limit: 30)` | Upstream received `limit=100` (capped; 300 would exceed mcpproxy-go's hard limit) |

All three tests use an in-process mock upstream that tallies incoming `params.arguments.limit` per call, so the bumped value is verified at the wire, not just at the helper boundary.

**Helper is pure:**
- Returns the input body unchanged when `limit` is absent or non-numeric (respects caller intent)
- Shallow-clones with spread — never mutates the input object
- Floors multiplier at `1` so misconfiguration can't shrink below the user's requested limit

## Compatibility

- Default multiplier `3` is on by default. No opt-in required.
- Set `SHIM_RETRIEVE_OVERREQUEST_MULTIPLIER=1` to disable the bump entirely (shim reverts to v1.6.1 behavior with dedup still active).
- Upstream `limit` is hard-capped at 100 regardless of multiplier — never exceeds mcpproxy-go's server-side limit validation.
- `compactRetrieveTools` is unchanged; it still slices to the original user `limit` after dedup, so user-facing response size is unaffected by the bump.
