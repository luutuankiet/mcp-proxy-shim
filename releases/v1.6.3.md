# v1.6.3 — Dedup respects mount/server identity

**Released:** 2026-04-14
**Type:** Correctness fix (backward-compatible, default-on, env kill-switch unchanged)
**Audience:** Anyone running v1.6.1 or v1.6.2 against a fleet with multiple upstream MCP servers that expose tools with the same trailing name (fs-mcp mounts, BigQuery instances, Looker instances, or anything else where two upstream servers ship the same underlying binary)

---

## TL;DR

v1.6.1 introduced M1 dedup with a dedup key that only looked at the last `__`-separated segment of a tool name. The intent was to fold the "same tool replicated across multiple hosts" case into one row. The effect was more aggressive: **any two tools whose trailing name segment + description + annotations + schema were byte-identical folded together — even when they represented different filesystems, different databases, or different Looker projects.**

On a fleet with 9 fs-mcp mounts (each a different filesystem root), all 9 `read_files` variants collapsed into one canonical entry. The `servers: [...]` metadata reported proxy-gateway identity (`utils`, `thinkpad`, `pi`), not mount identity — so the distinct filesystems disappeared from discovery. Same mechanic quietly applied to BigQuery and Looker fleets: `bq-one-prod__execute_sql` and `bq-doctor-anywhere-da__execute_sql` would have folded, routing subsequent calls to whichever instance won the arrival race.

v1.6.3 changes the dedup key to hash the **full mount-qualified tool name** (minus any leading `server:` routing prefix). Different upstream servers → different names → different keys → no spurious fold. Same mount reachable via multiple proxy gateways → same name → same key → still folds correctly. Mount identity is preserved, the legitimate cross-gateway fold case still pays out.

---

## Why This Release Exists

A live fleet test of v1.6.2 surfaced the problem. Calling `retrieve_tools("query_duckdb query_jq query_yq", limit: 8)` against a fleet of 9 fs-mcp mounts returned a row labeled `gce_at_slash__query_yq` with `servers: ["utils", "thinkpad", "pi"]` — but `gce_at_slash` is mounted only on `utils`; `thinkpad` and `pi` host completely different filesystems (nas, dell, personal, pi_at_slash). The `servers` list was telling the truth about which proxy gateways had folded in, but the single row hid eight other distinct filesystems that each ALSO had a `query_yq`. `hetzner_at_slash__query_yq`, `dell__query_yq`, `nas__query_yq`, etc. were all swept into that one canonical — invisible to the agent.

Routing was not broken (upstream mcpproxy-go still resolves each literal tool name correctly when called). But **discovery was materially wrong**: an agent that didn't already know the other mount names existed couldn't find them. And the fold wasn't fs-mcp-specific; the same logic applied equally to any fleet of multi-instance MCP servers that shared a schema.

The original design intent — fold the case where the same mount is exposed via multiple proxy gateways — is still correct and still valuable. The key just needed to be narrower.

## Highlights

| Change | What it does | Why it matters |
|---|---|---|
| `src/middleware.ts` dedup key | Full tool name (minus `server:` routing prefix) replaces the suffix-only key | Different mounts / different upstream servers with same-suffix tools stay as distinct entries. Mount identity, BigQuery project identity, Looker instance identity all preserved. |
| `src/middleware.ts` warn removed | The "small group (2)" drift signal is gone | It only made sense under suffix-only keying where tiny groups hinted the prefix-strip heuristic was misbehaving. With mount-qualified keys, a group of size N just means a mount is reachable from N proxy gateways — legitimate topology, no drift to warn about. |
| Cross-gateway fold preserved | Same mount name seen via utils/thinkpad/pi still collapses to one row | The legitimate dedup case — same filesystem, multiple access paths — still earns its token savings. |
| Tests flipped to two-axis fleet model | `DISTINCT_MOUNTS` (10 different mounts, must not fold) + `SHARED_MOUNT_TOOLS` (one mount, 3 gateways, must fold) | Fixtures now match the real fleet topology, not the v1.6.1 "10 replicated hosts" simplification. |
| Kill-switch test updated | Asserts shared-mount entries stay as 3 separate rows with `SHIM_DISABLE_DEDUP=1` | Still validates the kill-switch wires through the dedup path. |
| v1.6.2 slice assertion tightened | Excludes shim-local `proxy_admin` + `describe_tools` entries from the slice-cap check | Those are injected AFTER `compactRetrieveTools` slices; they always sit outside the user's limit budget. Assertion was spuriously over-counting them now that distinct mounts produce richer post-dedup results. |

## Before / After

**Before (v1.6.1..v1.6.2)** — `retrieve_tools("read_files")` against a fleet of 9 fs-mcp mounts on different filesystems:

```
pre-dedup: 9 distinct mount entries
            [alpha__read_files, bravo__read_files, ..., india__read_files]
dedup key: suffix="read_files" — same for all 9
result:   1 canonical row with
           name: <whoever_won_arrival_order>__read_files
           servers: [utils, thinkpad, pi]   (proxy gateways, not mounts)
agent sees: 1 tool. 8 filesystems hidden.   ❌
```

**After (v1.6.3)** — same call, same fleet:

```
pre-dedup: 9 distinct mount entries
dedup key: full_name — different for each of the 9
result:   9 separate entries preserved. Each keeps its own 'server' field.
agent sees: 9 tools. Every filesystem discoverable.   ✅
```

**Cross-gateway fold (still works):**

```
pre-dedup: 3 entries for shared_mount__read_files
            [server: utils, server: thinkpad, server: pi] — all same name
dedup key: full_name "shared_mount__read_files" — identical across the 3
result:   1 canonical entry with servers: [utils, thinkpad, pi]
agent sees: 1 tool, reachable via 3 gateways. Correct fold.
```

## Impact on Non-fs-mcp Fleets

The fix is NOT fs-mcp-specific. Any MCP fleet with multiple upstream servers that ship the same underlying binary benefits:

| Fleet pattern | v1.6.1..v1.6.2 behavior | v1.6.3 behavior |
|---|---|---|
| Multiple Looker instances (`looker-one-prod`, `looker-da`, `looker-joon`) with identical tool schemas | `get_explores`, `get_projects`, etc. folded into one canonical per tool — distinct Looker projects hidden in discovery | Each Looker instance's tools stay as separate entries. `looker-one-prod__get_explores` ≠ `looker-da__get_explores`. |
| Multiple BigQuery instances (`bq-one-prod`, `bq-doctor-anywhere-da`, `bq-joon-ken`) | `execute_sql`, `analyze_contribution` etc. folded across BQ instances — agents calling the canonical would silently route to whichever won arrival order | Each BQ instance stays separate. No cross-project confusion. |
| Multiple fs-mcp mounts on one or more proxy gateways | 9-way fold on this fleet, losing 8 filesystems from discovery | 9 distinct entries preserved. |

Before v1.6.3, the safety of enabling a second Looker or BigQuery instance depended on whether its tool schemas diverged enough to break the fold — which they often don't, since they come from the same binary. v1.6.3 makes multi-instance fleets safe.

## Verification

**Test suite:** 121 passed, 0 failed (`npm run test:daemon`). The fleet fixture was rewritten to model the real two-axis topology:

- **Axis 1 — distinct mounts, same gateway.** 10 different mount prefixes on a single proxy gateway. Assertion: stay as 10 separate entries, each with `server` singleton, no `servers` array.
- **Axis 2 — shared mount, multiple gateways.** One mount-qualified name (`shared_at_slash__read_files`) exposed via 3 proxy gateways. Assertion: fold into 1 canonical row with `servers: [utils, thinkpad, pi]`, singleton `server` removed, description prefix stripped.
- **Kill-switch test updated** to verify that with `SHIM_DISABLE_DEDUP=1`, the shared-mount entries return as 3 separate rows alongside the 10 distinct mounts.
- **v1.6.2 slice-cap assertion** tightened to exclude the always-injected shim-local entries (`proxy_admin`, `describe_tools`) which sit outside the slice budget.

**Live fleet sanity-check:** Tested against the production 9-mount fs-mcp fleet via the published shim. Calls to `retrieve_tools("query_duckdb query_jq query_yq", limit: 8)` now return per-mount entries instead of cross-mount folds, confirming mount identity at the wire.

## Compatibility

- Default behavior of `retrieve_tools` and `describe_tools` is unchanged for fleets with at most one instance per upstream MCP server type — such fleets had nothing to over-fold.
- Fleets that saw fewer tool rows than mounts on v1.6.1 or v1.6.2 will now see one row per distinct mount/instance. The row count per `retrieve_tools` call may grow.
- The synthetic 89.7% byte-reduction headline from v1.6.1 only holds for the cross-gateway fold case, which is narrower than the cross-mount fold it was erroneously catching. Real-world savings are now proportional to actual cross-gateway topology — smaller on most fleets, but *correct*.
- `SHIM_DISABLE_DEDUP=1` still disables all dedup, including the cross-gateway fold.
